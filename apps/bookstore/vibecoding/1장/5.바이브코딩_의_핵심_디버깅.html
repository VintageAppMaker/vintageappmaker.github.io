Debugging Masterclass

디버깅은 단순히 '고치는 것'이 아닙니다. AI에게 논리적으로 설명하여 정확한 해답을 이끌어내는 고도의 '소통의 기술'입니다.

왜 디버깅 프롬프트가 필요한가요?

AI는 사람처럼 상황을 '짐작'하지 못합니다. "안 돼, 고쳐줘"라고만 하면 AI는 엉뚱한 코드를 전체적으로 다시 짜버려 오히려 더 큰 버그를 만들 수 있습니다. 정확한 작업 지시서가 있어야 AI도 정확하게 고칠 수 있습니다.

RESOLVING...

디버깅의 핵심: "결과 중심으로 말하기"

나쁜 사례 (Vague)
"버튼이 안 눌려. 확인해줘."
정보 제공: (메시지 없음)
좋은 사례 (Specific)
"'할 일 추가' 버튼을 눌렀는데 리스트에 글자가 안 떠."
"개발자 도구 콘솔에 'ReferenceError: items is not defined'라고 떠."

성공적인 디버깅을 위한 5단계 공식

1

현상 설명

지금 어떤 동작이 안 되는지 결과 위주로 말합니다.

2

에러 공유

브라우저 개발자 도구(F12) 콘솔 창의 빨간색 메시지를 복사해 붙여넣습니다.

3

의도 재확인

원래 만들려던 기능의 순서(알고리즘)를 다시 알려줍니다.

4

부분 수정 지시

"전부 다 고치지 말고, 특정 부분만 수정해줘"라고 범위를 좁힙니다.

5

검증 요청

"수정한 코드가 기존의 다른 기능에 영향을 주지는 않니?"라고 물어봅니다.

버그를 예방하는 '바이브 코딩' 3대 수칙

로그 출력 지시

"모든 실행 과정을 `console.log`로 출력하는 코드를 짜줘."

데이터 구조 명시

데이터는 `JSON` 형태로 관리하고, `localStorage`에 저장해줘.

F12

개발자 도구 활용

버그가 나면 **F12**를 눌러 콘솔(Console) 탭을 확인하는 습관을 들입니다.

실전 디버깅 프롬프트 예제

Prompt_Example.md

// AI에게 보내는 메시지

"방금 준 코드에 문제가 있어."

현상: '삭제' 버튼을 클릭해도 화면에서 항목이 지워지지 않아.

에러 내용: 콘솔창을 보니 Uncaught TypeError: Cannot read property 'id' of undefined라고 나와.

추측: 삭제 함수에서 데이터를 찾는 로직에 문제가 있는 것 같아.

지시: 데이터 타입을 확인해보고, 삭제 후에 리스트가 자동으로 다시 그려지도록(렌더링) 해당 함수 부분만 수정해줘."

왜 프로그래밍 기초(JS)를 알아야 할까?

AI가 코드를 대신 짜주더라도, "어디가 틀렸는지" 판단하고 "어떻게 고치라고 지시"하는 것은 결국 사람의 몫이기 때문입니다.

01

변수와 데이터 타입

`let`, `const`의 차이나 데이터가 문자인지 숫자인지 이해하면 AI에게 더 정확한 지시를 내릴 수 있습니다.

02

비동기 처리(async/await)

서버 통신이나 타이머 관련 버그는 이 개념을 알 때 해결 속도가 5배 이상 빨라집니다.

요약하자면: 디버깅 프롬프트는 AI에게 주는 "업그레이드된 작업 지시서"입니다.

구체적으로 말할수록 버그는 빨리 사라집니다.
