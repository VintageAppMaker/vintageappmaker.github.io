일치 연산자 (===) 가이드

동적 타이핑의 함정을 피하고, 더 안전하고 예측 가능한 코드를 작성하기 위한 핵심 비교 연산법을 알아봅니다.

1. 필요한 이유

JavaScript는 동적 타이핑(Dynamic Typing) 언어입니다. 변수의 타입을 실행 중에 결정하므로 편리하지만, 비교 시 암묵적 형변환(Implicit Type Conversion)이라는 복병을 만납니다.

"대충 비슷하면 같다"라고 판정하는 일반 비교 연산자(==)는 서비스의 치명적인 논리 오류(결제, 인증 등)를 야기할 수 있습니다.

!

논리적 오류 방지: 숫자 5와 문자 "5"를 동일하게 취급하면 데이터의 무결성이 깨집니다.

비교 연산의 메커니즘 도식

Loose Equality
==
값(Value)이 같은가?
같으면 TRUE 반환
Strict Equality
===
값(Value)이 같은가?
타입(Type)도 같은가?
둘 다 만족해야 TRUE
연산자 값 비교 타입 비교 결과 판정
== ✅ ❌ 하나라도 맞으면 같다고 봄 (느슨한 비교)
=== ✅ ✅ 둘 다 일치해야 같다고 봄 (엄격한 비교)

핵심 예제 코드 

// 1. 숫자와 문자의 비교
console.log(5 == "5");    // true  (값이 비슷하니 OK)
console.log(5 === "5");   // false (Number vs String)

// 2. 특수 데이터의 비교
console.log(null == undefined);   // true  (의미가 비슷함)
console.log(null === undefined);  // false (타입이 다름)

// 3. 불리언(Boolean) 비교
console.log(0 == false);  // true  (0은 Falsy값)
console.log(0 === false); // false (Number vs Boolean)

심층 분석

5 === "5" 가 false인 이유

왼쪽 5는 Number, 오른쪽 "5"는 String입니다. 일치 연산자는 이 '데이터의 성격'을 따지기에 '다름'으로 결론 내립니다.

0 === false 가 false인 이유

JS에서 0은 거짓으로 간주(Falsy)되지만, 엄밀히 말해 0은 숫자이고 false는 논리값입니다. 타입의 벽을 넘지 못해 안전하게 처리됩니다.

💡 강사의 팁

"현업에서는 예외적인 상황을 제외하고는 항상 일치 연산자(===)를 사용하는 것을 권장합니다."

그래야만 예측하지 못한 형변환으로 인한 버그를 원천 봉쇄할 수 있습니다. (참조: 매뉴얼 35페이지)
